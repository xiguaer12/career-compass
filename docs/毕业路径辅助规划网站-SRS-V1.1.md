## 1. 引言

### 1.1 编写目的
本文档用于定义“本校本科应届毕业生未来路径规划平台”V1.1 的完整软件需求规格，作为概要设计、详细设计、开发、测试、验收和变更控制的唯一需求基线。

### 1.2 产品范围
本产品为网站形态，面向本校本科应届毕业生，围绕考公、考研、就业三条路径，支持学生完成“自我评估、路径比较、方案选择、行动规划、信息获取、经验参考”六步闭环。平台首版仅包含学生与管理员两个正式角色。

### 1.3 预期读者
- 产品经理
- 交互与视觉设计师
- 前后端开发工程师
- 测试工程师
- 运营与内容审核人员
- 校方项目管理人员

| 术语 | 定义 |

## 2. 总体说明

### 2.1 产品定位
平台定位为“本校本科应届毕业生未来路径规划平台”，以学生前台使用为主，以管理员后台运营和审核为辅。平台兼顾工具性与内容性，不承接报名、缴费、签约、培训购买等交易行为，不承诺上岸、录取或就业结果。

### 2.2 业务目标
- OBJ-001 在一个平台内完整覆盖考公、考研、就业三条路径，不偏废其中任一路径。
- OBJ-002 帮助学生完成从自我评估到行动规划的六步闭环。
- OBJ-003 通过 AI 辅助分析、图表、经验帖、问答、公告、专题内容以及自动采集后审核发布的权威公开信息，降低学生信息不对称。
- OBJ-004 使管理员可独立维护内容、图表、模板、标签、数据源、解析规则与审核规则，不依赖研发每次手工改库或改代码。

### 2.3 产品视角
系统由学生前台与管理员后台两部分组成：
- 学生前台负责注册登录、认证建档、问卷测评、AI 报告、路径内容浏览、图表浏览、社区互动、消息接收与个人资料维护。
- 管理员后台负责用户管理、内容审核、内容维护、图表维护、标签管理、数据源与采集任务管理、AI 配置、运营看板与审计日志查看。

### 2.4 用户类别与特征
| 用户类别 | 特征 | 主要目标 |
| --- | --- | --- |
| 游客 | 未登录用户，可浏览公开信息 | 了解产品、浏览公开内容、决定是否注册 |
| 待补档学生 | 已完成邮箱验证但未完成基础档案 | 完成建档，进入测评流程 |
| 待问卷学生 | 已完成基础档案但未完成深度问卷 | 完成问卷，获取首份 AI 报告 |
| 已完成引导学生 | 已完成档案与问卷 | 使用工作台、浏览内容、互动、获得个性化建议 |
| 管理员 | 学校运营/辅导/内容审核人员 | 维护数据、配置模板、审核内容、运营分析 |

### 2.5 运行环境
- ENV-001 系统形态为网站。
- ENV-002 学生前台必须适配桌面端与手机浏览器。
- ENV-003 管理员后台以桌面端浏览器为主要使用环境。
- ENV-004 兼容范围见 6.7 兼容性要求。

### 2.6 设计约束
- CON-001 角色冻结为学生与管理员，不引入导师、企业、培训机构等新角色。
- CON-002 数据来源冻结为校内维护数据与公开权威数据；公开权威数据既可人工维护，也可通过系统自动抓取。
- CON-003 自动抓取仅限管理员配置并批准的公开权威来源，且抓取结果必须经过人工审核后方可前台发布。
- CON-004 AI 仅作为顾问助手，不替代最终决策。
- CON-005 当前文档不展开数据库结构、接口协议、部署方案和模型选型。

### 2.7 范围边界
#### 2.7.1 纳入 V1.1 的范围
- 学生注册登录与认证建档
- 深度问卷与 AI 报告
- AI 报告追问
- 个人工作台
- 三条路径内容页
- 图表中心
- 社区与问答
- 消息中心与个人中心
- 管理员后台运营维护能力
- 公开权威数据自动抓取、解析入库、人工审核发布

#### 2.7.2 不纳入 V1.1 的范围
- 报名、缴费、签约、培训付费等交易行为
- 简历投递、岗位申请、研究生报名、公务员报名等外部业务闭环
- 导师指导、企业招聘、培训机构入驻
- 小程序、原生 App、企业端与导师端
- 毕业后一年以上的长期跟踪
- 对需登录、需付费、受版权或协议限制的数据源进行抓取

## 3. 业务对象与业务规则

### 3.1 关键业务对象
| 对象 | 关键字段 | 主要规则 |
| --- | --- | --- |
| 学生账户对象 | 邮箱、密码、验证码状态、姓名、学号、学院、专业、毕业年份、手机号、昵称、头像、隐私设置、协议同意状态、账号状态、注册时间、最后登录时间 | 邮箱唯一；邮箱域名必须命中后台配置名单；前台默认不公开姓名、学号、手机号 |
| 测评问卷对象 | 问卷版本、题目分组、题目类型、必填规则、草稿状态、完成状态、更新时间、提交时间、输入快照 | 支持草稿与完成态；完成后不得直接修改历史快照；模板需版本化 |
| AI 报告对象 | 报告版本、生成时间、输入快照、三路径评分、推荐排序、理由说明、风险提示、30/60/90 天计划、资源建议、免责声明、追问记录、生成状态 | 每份报告必须覆盖三路径；保留生成版本与输入快照；失败任务允许重试 |
| 图表对象 | 图表类型、标题、数据项、筛选条件、口径说明、来源说明、更新时间、可见范围、展示位置、状态 | 所有公开图表必须显示来源、统计口径、更新时间 |
| 数据源对象 | 来源名称、来源类型、来源地址、抓取频率、适用路径、启停状态、来源可信级别、最后抓取时间 | 仅允许配置已批准的公开权威来源；来源变更需留痕 |
| 采集数据对象 | 采集任务号、来源对象、抓取时间、原始链接、解析结果、关联标签、审核状态、发布时间、失败原因 | 自动抓取结果必须先进入待审核状态，审核通过后才可转为前台公开内容或图表数据 |
| 社区内容对象 | 内容类型、标题、正文、标签、作者展示方式、审核状态、互动数据、举报状态、最佳回答标记、创建时间、更新时间 | 支持经验帖与问答；问答支持匿名发布；内容需可审核与可追溯 |
| 后台配置对象 | 公告、推荐位、标签、问卷模板、报告模板、图表模板、敏感词规则、系统消息模板、路径专题内容 | 所有配置项均应支持后台独立维护并记录版本或变更日志 |

### 3.2 关键数据校验规则
| 编号 | 规则 |
| --- | --- |
| DR-001 | 学生邮箱必须匹配管理员配置的学校域名名单，例如 `@xxx.edu.cn`；域名名单可配置，不允许写死在代码验收标准中。 |
| DR-002 | 同一邮箱仅允许注册一个有效学生账户。 |
| DR-003 | 密码长度应为 8 至 20 位，且至少包含字母、数字、特殊字符中的任意两类。 |
| DR-004 | 姓名为 2 至 20 个字符。 |
| DR-005 | 学号为 6 至 20 位字母或数字，且在有效学生账户中唯一。 |
| DR-006 | 毕业年份必须属于后台配置的有效毕业年份范围。 |
| DR-007 | 手机号应符合中国大陆 11 位手机号格式。 |
| DR-008 | 昵称长度应为 2 至 20 个字符，可与真实姓名不同。 |
| DR-009 | 头像文件大小不得超过 5 MB，格式限定为 JPG、JPEG、PNG。 |
| DR-010 | 经验帖标题长度应为 5 至 50 字；正文长度应为 50 至 10000 字。 |
| DR-011 | 问答标题长度应为 5 至 50 字；问题正文长度应为 10 至 3000 字；回答正文长度应为 5 至 3000 字。 |
| DR-012 | 单条内容标签数量应为 1 至 5 个。 |
| DR-013 | 图表标题长度应为 2 至 50 字；每个图表必须具备来源说明、统计口径和更新时间。 |
| DR-014 | AI 报告三路径评分范围为 0 至 100 的整数。 |
| DR-015 | 比例类图表展示精度为小数点后 1 位；计数类数据展示为整数；由于四舍五入造成的总和误差允许在 99.9% 至 100.1% 之间。 |
| DR-016 | 自动抓取来源必须为管理员配置的白名单来源，且需记录来源名称、来源地址和适用路径。 |
| DR-017 | 每条自动抓取数据必须保留原始链接、抓取时间、解析时间和来源标识。 |
| DR-018 | 自动抓取结果在审核通过前不得对学生前台公开。 |
| DR-019 | 采集失败记录必须保留失败原因和失败时间，便于重试与审计。 |

### 3.4 核心业务规则
- BR-001 每份 AI 报告都必须同时覆盖考公、考研、就业三条路径。
- BR-002 平台首页、路径页、图表中心、社区内容组织均不得将任何一条路径排除在产品主导航之外。
- BR-003 新用户首次登录后必须完成基础档案和深度问卷，未完成前仅可浏览公开内容，不可发帖、提问、回答、收藏个性化报告或进入完整工作台。
- BR-004 平台不承接交易行为，不输出录取、上岸、就业结果承诺。
- BR-005 所有公开图表必须展示数据来源、统计口径和更新时间。
- BR-006 所有 AI 页面必须固定展示“仅供辅助决策”免责声明。
- BR-007 所有社区内容必须具备举报入口，且管理员必须可审核与处置。
- BR-008 前台默认不展示真实姓名、学号和手机号；匿名问答前台不展示昵称和头像，但后台必须可追溯真实发布者。
- BR-009 历史问卷快照、报告快照和关键审核结果必须可追溯，不因模板升级而被覆盖。
- BR-010 后台维护数据的日常更新不应依赖研发修改代码或直接改库。
- BR-011 自动抓取仅能面向已批准的公开权威来源执行，不得抓取受限登录、付费、版权受限或校方禁止的数据源。
- BR-012 自动抓取结果必须先进入待审核池，只有管理员审核通过后才能发布到首页、路径页、图表中心或专题内容。
- BR-013 自动抓取发布的数据必须展示来源名称、原始链接或来源说明、抓取时间/更新时间与统计口径。

## 4. 外部接口与展示需求

### 4.1 通用界面要求
- UI-001 前台页面必须同时适配桌面端与手机浏览器。
- UI-002 所有页面均应提供明确的加载中、空状态、错误状态和成功反馈状态。
- UI-003 所有涉及身份信息的前台展示均应遵守隐私设置，默认不公开敏感字段。
- UI-004 所有 AI 报告页与 AI 追问页必须在首屏可见区域展示免责声明。
- UI-005 所有图表卡片和图表详情页必须可见来源、统计口径、更新时间。
- UI-006 社区内容详情页必须展示内容标签、审核后可见状态下的互动信息、举报入口与时间信息。
- UI-007 管理员后台页面必须显示操作结果反馈，并对高风险操作提供二次确认。

| 首页 | 产品简介、三路径入口、热门图表、精选经验帖、公告、常见问题、注册登录入口、“开始测评”入口 |

### 8.4 自动抓取与审核发布规则
| 规则编号 | 条件或对象 | 输出要求 |
| --- | --- | --- |
| CRW-001 | 数据源配置 | 仅允许管理员配置公开权威来源白名单，并指定来源名称、来源地址、适用路径、抓取频率和启停状态 |
| CRW-002 | 抓取任务触发 | 支持定时触发和手动触发，两种方式均需记录任务日志 |
| CRW-003 | 抓取结果入库 | 每条结果必须记录原始链接、抓取时间、解析结果、来源标识和当前审核状态 |
| CRW-004 | 发布前审核 | 抓取结果在管理员审核通过前不得在前台公开 |
| CRW-005 | 审核操作 | 管理员至少可执行通过发布、驳回、编辑后发布和重新抓取 |
| CRW-006 | 发布展示 | 已发布抓取数据必须显示来源说明、原始链接或来源描述、抓取时间或更新时间 |
| CRW-007 | 失败处理 | 抓取失败或解析失败时必须保留失败原因，并允许重试 |
| CRW-008 | 合规边界 | 不得抓取需登录、需付费、法律法规禁止或学校禁止的数据源 |

## 10. 需求冻结与变更控制

### 10.1 冻结原则
- CHG-001 本文档作为 V1.1 的唯一需求基线，评审通过后冻结。
- CHG-002 冻结前必须完成产品评审，至少确认产品目标、角色范围、模块边界、验收标准、数据来源边界和非功能要求。
- CHG-003 冻结后不得以口头方式直接向开发新增需求。

### 10.2 变更分类
| 分类 | 说明 | 处理原则 |
| --- | --- | --- |
| 高影响变更 | 新增角色、新增终端、新增交易功能、扩展到毕业后长期跟踪、引入无需人工审核的自动发布、扩展到非权威或受限数据源等 | 不并入 V1.1，进入后续版本评审 |
| 一般变更 | 文案、标签、展示顺序、部分后台配置项、已批准来源白名单调整 | 可通过变更单评估后纳入，不得改变业务流程和验收边界 |

### 10.3 升级兼容要求
- UGR-001 后续版本升级不得破坏 V1.1 历史问卷、历史报告和历史图表说明的可读性。
- UGR-002 模板、标签和配置项的升级必须保留版本信息或变更留痕。
- UGR-003 涉及新增角色、终端、交易能力、取消人工审核直接自动发布、扩展到非白名单数据源或服务对象范围扩大时，必须作为新版本立项，不得按一般变更处理。
